# `src/site/` — Maven Site descriptor

Holds the [Maven Site](https://maven.apache.org/plugins/maven-site-plugin/)
configuration used by `mvn site` to generate a static HTML report bundle
(Surefire/Failsafe/JaCoCo/SpotBugs/PMD/Checkstyle/OWASP) under
`target/site/`.

## Files

| File       | Role                                                                                                                                                                           |
| ---------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `site.xml` | Site descriptor — declares the skin (`maven-fluido-skin` 2.0.0-M11, required since Doxia 2.0 to avoid "skin cannot be null" errors) and customises the left-nav menu (adds a manual link to the OWASP Dependency-Check HTML report). |
| `README.md`| This file.                                                                                                                                                                    |

## 🚧 Migration under consideration

**This directory — and the entire Maven Site integration — may be deprecated
in the medium term.** The Maven Site plugin has been slowly rotting (see the
"skin cannot be null" regression forced on us by Doxia 2.0, the Fluido skin
being at `2.0.0-M11` for months, PMD/Checkstyle integration bugs) and
Jeremy Long's `dependency-check-maven` already refuses to declare itself in
the `<reporting>` lifecycle because of the reporting API changes in Maven 4.

### Alternative already discussed

Serve the individual HTML reports directly (Surefire, JaCoCo, SpotBugs, PMD)
from the Angular UI's Quality page, the same way OWASP and Pitest are
served today: each report is a self-contained HTML file generated by its
own plugin at `verify` time, copied into a known path under
`target/reports/`, and exposed via the `/actuator/quality` endpoint. The
UI already renders them in an iframe or new tab.

Benefits:
- No Doxia, no skin, no site lifecycle — one less moving part.
- Report lifecycle is decoupled — broken skin doesn't block the report.
- Faster `mvn verify` (site generation adds ~30-60 s).

Until the migration happens, this folder stays and keeps `mvn site`
buildable. When it lands, this README + `site.xml` + the whole `<reporting>`
section in `pom.xml` can be deleted.

## Related

- `pom.xml` — `<reporting>` section + `maven-site-plugin` config.
- `config/pmd-ruleset.xml`, `config/spotbugs-exclude.xml`, `config/owasp-suppressions.xml` — consumed by the reports rendered into the site.
- `src/main/resources/META-INF/build-reports/` — the same artifacts, but cached and served from the JAR at runtime via `/actuator/quality`.
